The Quest Giver

What is bili‑core?

An open‑source shared library for building AI agents (chatbots that can reason, call tools, and remember conversations).

  • Built and maintained here at MSU Denver. You can read every line.
  • Standing on the shoulders of LangChain and LangGraph (industry tools for wiring up LLM apps).
  • It's a Python library organized into three big subsystems (Python packages), the three realms we're about to explore.
The Quest Giver

Why a shared library?

One library, many kingdoms. bili‑core holds the pieces every AI app needs, so no team rewrites them.

The Sustainability Hub is one kingdom built on it. So is your future chatbot.

The rule of the land Fix a bug in the library and every kingdom gets the fix. Great power, great responsibility.

Act I • The Map

The lay of the land

bili/
├ iris/          # single-agent realm
├ aether/        # multi-agent realm

└ *_app.py       # entry points

Three realms do the AI work; the rest is shared support (auth, utilities, prompts, interfaces).

The State that every conversation carries lives in bili/utils/langgraph_utils.py. Hold that thought, we return to it.

Act I • The Map

Your provisions (dependencies)

  • LangChain 1.0 + LangGraph 1.0: the backbone for wiring LLM steps together.
  • Provider SDKs: OpenAI, Azure, Google Vertex, AWS Bedrock, Hugging Face, local models.
  • Databases: Postgres & Mongo drivers, to remember conversations.
Heavy armor, optional torch and tensorflow ship too, but you only wield them for local models. Use a hosted provider and they just sit in your pack.

All declared in requirements.txt. Installed with a plain pip install .

Act I • A Word on Magic

LangGraph, in one breath

If you've met a state machine or a flowchart in class, you already understand this.

  • Node: one step that does a job (add a persona, call the model, save a title).
  • Edge: an arrow from one step to the next.
  • State: a bag of data (the messages, a summary, the title) passed along and updated at each node.
START
  → add_persona
  → call_the_model
  → save_the_title
END

bili‑core's whole job is building and running these graphs for you.

Act II • The Three Realms

Meet the three realms

Each “realm” is really a subsystem, a self‑contained Python package inside the library.

Realm I

IRIS

One agent, done well. The single chatbot: model + tools + memory.

Realm II

AETHER

Many agents, working together. A team of specialists from a YAML recipe.

Realm III

AEGIS

Attacking the team, on purpose. Security testing for multi‑agent systems.

The Hub you'll see in Act IV lives entirely in the first realm. The other two are the frontier.

Realm I • IRIS

IRIS: the single agent

Interactive Reasoning and Integration Services

Give it a model, some tools, and a memory, and it runs a conversation. This is the realm your take‑home quest lives in.

  • 60+ models across 6 providers, chosen by a single string.
  • A tidy pipeline of nodes handles persona, reasoning, memory, and cleanup.
  • Conversations persist automatically, so users can come back tomorrow.
Realm I • IRIS

Four steps to a living chatbot

# 1. choose a brain
model = load_model(model_type="remote_openai", model_name="gpt-4o-mini")

# 2. hand it some tools (or none)
tools = initialize_tools(active_tools=[], tool_prompts={})

# 3. assemble the graph (memory is automatic)
agent = build_agent_graph(node_kwargs={"llm_model": model, "tools": tools, ...})

# 4. speak to it
invoke_agent(agent, "Hello, who are you?", thread_id="user1")

That's the entire skeleton. Everything else is decoration.

Realm I • IRIS

What happens inside invoke

Your message walks a line of nodes, each doing one small job:

START
 → add_persona_and_summary   # who am I + what happened so far
 → inject_current_datetime    # what time is it now
 → react_agent               # ⭐ the actual thinking + tool use
 → update_timestamp           # bookkeeping
 → trim_summarize             # keep memory from overflowing
 → normalize_state            # tidy up
END

Remember this whole pipeline (the full START→END line‑up). In Act IV, the Hub splices its own nodes right into it.

Realm I • IRIS

The State: the party's shared pack

Every node reads from and writes to one shared object. It carries:

  • messages: the conversation
  • summary: the running recap
  • owner: whose chat this is
  • title, tags: metadata
  • timestamps, llm_config, and more
Why it matters

Nodes never pass arguments to each other directly. They just update the shared State, and the next node sees it. Additive updates: a node returns only the fields it changed.

Realm I • IRIS

Try it with no code: the Streamlit app

bili‑core ships a ready‑made Streamlit chat app (port 8501). It puts an IRIS agent behind a friendly web page, so you can experiment without writing a single line.

  • Pick and configure any of the 60+ models from a menu.
  • Swap the model mid‑conversation, then watch how the same chat behaves on a different LLM.
  • A lovely way to feel how model behavior adapts.
📜 A bit of Hub history

This was the original app the team built, to figure out which LLM was best suited for the Sustainability Hub.

Realm II • AETHER

AETHER: a party of agents

Agent Ecosystems for Testing, Hardening, Evaluation, and Research

Where IRIS is one adventurer, AETHER is a whole party: several agents, each its own, working together. You decide who's in the party and what each one does.

The roles are free‑form and up to you. A moderation team might be a Judge, a Reviewer, and a Policy Expert, but a research team or a code‑review team would look completely different.

  • You describe the team in a YAML file. No graph code required.
  • AETHER compiles that recipe into a runnable LangGraph workflow.
  • Every agent can have its own model, tools, and personality.
Realm II • AETHER

From recipe to running team

Define  →  Configure  →  Validate  →  Compile  →  Execute

In code, that's a clean straight line:

config   = load_mas_from_yaml("team.yaml")
compiled = compile_mas(config)   # checks first!
result   = execute_mas(config, {...})
A kind guardrail

Validate runs 13 safety checks before anything executes (orphaned agents, dead ends, cycles). Errors stop the build; warnings just advise. It fails early, not mid‑conversation.

Realm II • AETHER

Choose your party's formation

One field, workflow_type, picks how the agents are wired:

  • sequential: a relay line, one after another
  • supervisor: a boss routes work to specialists
  • consensus: peers vote until they agree
  • hierarchical: tiers that escalate upward
  • parallel: everyone at once, then merge
  • custom: your own conditional arrows
Nice touch A single AETHER agent that also has tools is built the same way as an IRIS agent. AETHER stands on IRIS's shoulders.
Realm II • AETHER

A real recipe: content moderation

workflow_type: supervisor
entry_point: judge
agents:
  - agent_id: judge          # the boss
    is_supervisor: true
  - agent_id: content_reviewer
  - agent_id: policy_expert
  - agent_id: appeals_specialist

A flagged post arrives at the Judge, who calls specialists as needed and synthesizes a final verdict.

This exact file ships in the repo as supervisor_moderation.yaml. It's the very system a student is exploring for the Hub's own moderation someday.

Realm III • AEGIS

AEGIS: the loyal adversary

Adversarial Evaluation and Guarding of Intelligent Systems

If AETHER builds the multi‑agent machine, AEGIS attacks it on purpose, and measures what breaks.

  • Injects adversarial payloads (prompt injection, memory poisoning, impersonation…).
  • Tracks how an attack propagates from one agent to the next.
  • Built by MSU Denver's Cybersecurity Research group. This is real security research.
Realm III • AEGIS

Three tiers of “did it work?”

Tier 1

Structural

Did the attack even run without crashing the framework? Free, always on.

Tier 2

Heuristic

Does the output look influenced? Fast string checks. A useful proxy.

Tier 3

Semantic

An independent LLM judge confirms it. The real verdict.

The research gem The gap between Tier 2 (“looks bad”) and Tier 3 (“is bad”) is itself a finding worth studying.

Act III • Provisions

What does it take to run?

Less than you'd fear.

  • One provider key is enough. An OPENAI_API_KEY gets you going.
  • No database required. With no DB configured, memory falls back to in‑process storage automatically.
  • Add Postgres or Mongo later only when you want conversations to survive a restart.
Checkpointer priority Postgres → Mongo → in‑memory. It picks the best one you've configured, with a safe fallback.

Act IV • The Kingdom

The golden rule: extend, don't modify

The library owns the data

The title and tags fields live on bili‑core's shared State, because every app might want them.

The app owns the behavior

But how the Hub computes a title or tags is its own business, so that logic lives in the Hub, as custom nodes.

Generic data in the library. Specific behavior in the app. That one idea explains the whole design.

Act IV • The Kingdom

The Hub adds two nodes of its own

  • compute_title: names the chat from the first message (once).
  • update_tags: pulls topic tags every few messages.

It copies bili‑core's default pipeline and splices these in, never mutating the shared original.

# the whole extension trick
compute_title_node = partial(
    Node, "compute_title",
    build_compute_title_node)

A custom node is just bili‑core's own Node with the app's function inside.

Act IV • The Kingdom

The Hub's assembled pipeline

Same IRIS backbone, with two Hub nodes woven in (⭐):

START
 → add_persona_and_summary
 → inject_current_datetime
 → compute_title       # ⭐ Hub
 → react_agent
 → update_timestamp
 → update_tags         # ⭐ Hub
 → trim_summarize
 → normalize_state
END

Merged registry: {**GRAPH_NODE_REGISTRY, **ENGINE_NODE_REGISTRY}. Library nodes ∪ Hub nodes.

Act IV • The Kingdom

The Hub brings its own tools, too

The Hub extends bili‑core two ways: custom nodes (which we just saw) and custom tools. Its star tool, knowledge_search, is a real‑world piece of RAG.

RAG, plainly

Retrieval‑Augmented Generation: let the agent look things up in your documents before it answers, instead of guessing from memory alone.

# knowledge_search, in spirit
q     = embed(question)          # question → vector
docs  = opensearch.search(q)     # find relevant docs
answer = agent(question, docs)   # answer, grounded

IRIS gives you the agent; add a retriever tool like this and you've got RAG. The library ships the hooks (embeddings + retriever tools); the Hub supplies the documents.

Act IV • The Kingdom

Three gates into the kingdom

  • Streamlit (8501): the quick UI, mostly bili‑core's prebuilt pages.
  • Flask (5001): REST API for auth and data.
  • WebSocket (3500): the real‑time chat, where the full agent is built.

A WebSocket message, end to end:

  1. Check the JWT + that the thread belongs to you
  2. build_agent_graph(...) with the spliced pipeline
  3. agent.invoke(...) runs the nodes
  4. Checkpointer saves the conversation automatically
  5. Reply streams back with title + tags
Act IV • A Guard at the Gate

One security idea worth stealing

Every conversation's key is {your_email}_{conversation_id}.

  • Before loading any thread, the Hub checks it starts with your email.
  • So you can never open someone else's chat by guessing an ID.
The takeaway A tiny ownership check, enforced at the door, prevents a whole class of attacks. Good design is often this humble.

Act V • Your Own Quest

Your first spell, in full

from bili.iris.loaders.llm_loader import load_model
from bili.iris.loaders.tools_loader import initialize_tools
from bili.iris.loaders.langchain_loader import build_agent_graph
from bili.iris.loaders.streaming_utils import invoke_agent

model = load_model(model_type="remote_openai", model_name="gpt-4o-mini")
tools = initialize_tools(active_tools=[], tool_prompts={})
agent = build_agent_graph(node_kwargs={
    "llm_model": model, "tools": tools,
    "persona": "You are a wise tavern keeper.",
    "summarize_llm_model": model,
})
print(invoke_agent(agent, "Greetings! What's on the menu?", thread_id="hero1"))
